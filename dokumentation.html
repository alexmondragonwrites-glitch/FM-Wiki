ARCHITEKTUR · VERSION 2

Vier zentrale Bauteile

Konfiguration

data/config.js enthält Stichtag, Saison, Navigation, Schema-Version und Footertext.

Manifest

data/manifest.js bestimmt die Reihenfolge von Basisdaten, aktuellen Ergänzungen und optionalen Profilen.

Loader

assets/data-loader.js lädt Datendateien sequenziell und verhindert unkontrollierte Parallelität.

Seitenshell

assets/site-shell.js erzeugt Navigation und Footer zentral für alle Hauptseiten.

DATENMODELL

Basis, aktueller Stand und optionale Profile

Spielerdaten werden nicht mehr als eine undurchsichtige Skriptkette verstanden. Das Manifest unterscheidet Basisdaten, aktuelle Fortschreibungen und optionale Profile wie bestätigte Zugänge für die nächste Saison.

  • Basisdaten: ursprüngliche Kaderobjekte und Geburtsdaten.
  • Aktueller Stand: datierte Korrekturen, Ergebnisse und Rollenfortschreibungen.
  • Optionale Profile: vorbereitete oder bestätigte Zugänge, die nur dort geladen werden, wo sie fachlich hingehören.

HISTORISCHE LOGIK

Chronik ohne Datenlawine

Datierte Ergänzungsdateien bleiben erhalten. Die Live-Seiten laden jedoch nur die im Manifest als aktuell markierten Schichten. Kaderarchive besitzen eigene Referenzdaten über data/seasons.js. Dadurch kann Saison 2041 beginnen, ohne den Abschlussstand 2040 umzuschreiben.

QUALITÄTSSICHERUNG

Ein Türsteher vor GitHub Pages

scripts/validate-data.mjs prüft den Bestand automatisch. Die GitHub Action läuft bei Änderungen auf main und bei Pull Requests.

  • JavaScript-Syntax
  • fehlende Manifest-Dateien
  • defekte lokale Links
  • doppelte Spieler-IDs
  • Pflichtfelder und Datumsformate
  • Navigationsziele

UPDATE-WORKFLOW

So wächst das Wiki

  1. Quelle erfassenScreenshot oder Export einem konkreten Datum und Bereich zuordnen.
  2. Datenebene wählenBasis, aktueller Stand, optionales Profil oder Saison-Snapshot.
  3. Manifest prüfenNur live relevante Schichten werden in den aktuellen Ladeweg aufgenommen.
  4. Renderer aktualisierenDarstellung direkt korrekt erzeugen, keine nachträglichen DOM-Reparaturen.
  5. Validierung bestehenSyntax, IDs, Links und Datumswerte müssen fehlerfrei sein.

TECHNIK

Statisch, aber nicht mehr improvisiert

Das Wiki läuft weiterhin ohne Server, Datenbank oder Build-System direkt auf GitHub Pages. Das ist für den aktuellen Umfang sinnvoll. Der Unterschied liegt im Rückgrat: zentrale Konfiguration, manifestgesteuerte Daten, gemeinsame Seitenshell und automatische Prüfung.
